## 1. 代码质量审查

### 1.1 必需检查项

- [ ] **类型提示完整性**
  - [ ] 所有公共函数/方法有类型注解
  - [ ] 函数返回值类型已标注
  - [ ] 类实例变量有类型注解

- [ ] **错误处理**
  - [ ] 异常被精确捕获（非 `except Exception`）
  - [ ] 关键操作有 try/except 保护
  - [ ] 错误信息有意义且包含上下文

- [ ] **代码复杂度**
  - [ ] 函数长度 < 100 行
  - [ ] 圈复杂度 < 10
  - [ ] 嵌套层级 < 4 层

### 1.2 推荐检查项

- [ ] **代码可读性**
  - [ ] 变量/函数命名清晰
  - [ ] 复杂逻辑有注释
  - [ ] 常量有命名

- [ ] **安全性**
  - [ ] 无硬编码凭证
  - [ ] 用户输入已验证
  - [ ] 无 SQL 注入风险

## 7. 审查流程

### 7.1 审查者职责

1. **完整性检查**: 确保清单所有项目已检查
2. **客观评价**: 基于事实而非偏好
3. **建设性反馈**: 提供具体改进建议
4. **及时响应**: 在 24 小时内完成审查

### 7.2 作者职责

1. **自检先行**: 提交前对照清单自检
2. **响应反馈**: 及时回应审查意见
3. **记录决策**: 重要决定记录在 PR 备注中

### 7.3 评分标准

| 级别 | 通过条件 |
|------|----------|
| **Approved** | 所有必需项通过 |
| **Request Changes** | 有必需项未通过 |
| **Comment Only** | 仅为建议性反馈 |

## 8. 常见问题处理

### 8.1 类型提示缺失

```
建议: 使用 Optional[T] 而非 T | None
建议: 使用 List[T] 而非 list[T]
建议: 添加函数返回类型注解
```

### 8.2 并发安全问题

```
建议: 使用 with self._lock: 保护临界区
建议: 使用 RLock 支持重入
建议: 避免在锁外访问共享资源
```

### 8.3 测试覆盖率不足

```
建议: 添加边界条件测试
建议: 添加异常路径测试
建议: 使用 pytest fixtures 复用测试数据
```
