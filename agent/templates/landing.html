Technical Walkthrough

详细技术解说：从用户输入到系统响应，WarmStudy 到底做了什么

这一部分专门为答辩和技术评审准备，不只是列技术名词，而是解释每一层为什么存在、它如何服务于青少年心理支持场景。

1. 统一入口层：App 与网关

学生端和家长端通过统一接口体系进入系统，网关层负责状态检查、路由治理、RAG 代理、后台入口与服务聚合，保证前端调用路径清晰、部署结构稳定。

Frontend App API Gateway 统一入口

2. 策略层：状态驱动而不是固定 Prompt

系统会综合用户角色、年龄、学段、近期打卡、测评状态、历史上下文与风险变化，动态调整输出风格和支持策略，让回答更贴近“这个人此刻的状态”。

角色差异 状态驱动 风险感知

3. 知识增强层：RAG 不是装饰

心理研究资料会先切片，再用 text-embedding-v3 生成向量存入数据库。用户提问后，系统先检索相关片段，再经重排筛选高质量证据，最后连同问题一起送入大模型生成回答。

  • 切片让知识更适合检索
  • Embedding 让语义可计算
  • 重排提升召回片段的相关性

4. 业务闭环层：评测、报告、绑定、提醒

WarmStudy 不是“聊完就结束”，而是围绕学生评测、AI 报告生成、家长查看、家校观察和风险提醒形成完整闭环，让系统从工具升级为支持体系。

  • 学生完成评测后生成专业性报告
  • 家长端获得实时查看与 AI 建议
  • 后台承接持续观察与运营管理

5. 为什么这条技术路线适合比赛

因为它不仅能展示“用了什么技术”，还能回答“为什么这样设计”。每个技术点都能映射到清晰的业务价值，这会比单纯展示聊天能力更有说服力。

Key Flows

把技术流程和业务流程都讲清楚

这部分不是纯技术示意，而是帮助评委迅速看懂“这个系统为什么成立”的关键证据。

RAG · 提问前

先把知识准备好，回答才有依据

心理资料被切分成片段，再通过 Embedding 转为向量并写入数据库，这一步决定了系统是否真正拥有可检索、可调用的知识底座。

RAG · 提问后

不是直接生成，而是先召回、再重排、再回答

系统先检索，再重排，再把高相关内容交给大模型，避免纯生成式回答脱离心理支持知识依据。

Business Flow

评测不是结束，而是支持链路的开始

学生完成评测后，系统会生成 AI 心理报告并同步给家长端，让测试结果变成可理解、可行动的支持信息。

State Awareness

理解状态，而不是只理解句子

通过情绪识别、打卡结果和历史上下文，WarmStudy 让智能体回答更贴近学生此刻的心理状态。
