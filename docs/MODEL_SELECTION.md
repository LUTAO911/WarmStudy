# WarmStudy Model Selection Guide

## 1. 这份文档解决什么问题

这个项目不是单纯追求“模型越强越好”，而是要在以下维度里找到平衡：

- 对话质量
- 情绪陪伴风格稳定性
- 危机识别的可靠性
- 响应速度
- 单次对话成本
- 长期运营总成本
- 数据与供应商可控性

## 2. 当前代码已经支持什么

从现有代码看，后端已经明确支持两类模型接入：

- 通义千问 / 阿里云百炼
- MiniMax

代码里也保留了进一步扩展更多模型提供商的空间，因此如果后续要接入 DeepSeek、豆包或 OpenAI，建议以“新增 provider 适配层”的方式做，而不是把判断逻辑继续堆进 `app.py`。

## 3. 选型原则

对于青少年心理陪伴场景，建议不要只看基准榜单，而要优先看以下四点：

1. 中文表达是否自然、温和、少机械味
2. 长对话中人设和语气是否稳定
3. 遇到危机词、求助词时是否容易失控或答偏
4. 每百万 Token 成本能否支撑规模化运营

## 4. 推荐模型分层

### 4.1 默认推荐

如果你们现在要做“能上线试点、成本可控、中文体验稳”的版本，建议：

- 主对话模型：`qwen-plus`
- 高价值场景升级模型：`qwen-max` 或 `MiniMax-M2.7`
- 低成本批处理 / 摘要 / 分类：后续接入 `deepseek-chat`

原因：

- `qwen-plus` 已经和当前项目代码契合度最高，接入成本最低
- `qwen-max` 适合高风险、高价值、高复杂度场景兜底
- DeepSeek 在成本上很有优势，适合做非实时或弱交互任务

### 4.2 不同阶段推荐

#### 比赛演示阶段

- 首选：`qwen-plus`
- 兜底：`qwen-max`

目标：

- 少改代码
- 保证中文回复自然
- 减少因多供应商切换带来的不确定性

#### 校内试点阶段

- 主模型：`qwen-plus` 或 `MiniMax-M2.5`
- 升级模型：`qwen-max` 或 `MiniMax-M2.7`
- 离线处理：`deepseek-chat`

目标：

- 压低成本
- 让实时陪伴与后台分析分层

#### 商业落地阶段

- 实时陪伴：稳定供应商双活
- 高风险转人工前分析：高质量模型
- 报告生成、周报、标签、摘要：低成本模型

目标：

- 成本结构清晰
- 供应商风险可控
- 关键链路可灰度切换

## 5. 结合当前项目的推荐组合

## 方案 A：最稳妥

- 默认模型：`qwen-plus`
- 升级模型：`qwen-max`

适合：

- 当前代码基础
- 需要快速交付
- 需要答辩时解释简单

优点：

- 与现有环境变量和代码最贴合
- 中文任务表现稳定
- 迁移成本最低

缺点：

- 如果高频长对话很多，成本会高于极致性价比方案

## 方案 B：性能与成本平衡

- 默认模型：`MiniMax-M2.5`
- 升级模型：`MiniMax-M2.7`
- 备用模型：`qwen-plus`

适合：

- 更在意响应速度与综合成本
- 后续会扩展语音、多模态或 Agent 能力

优点：

- 官方按量计费页已经给出了清晰文本计价
- 文本、语音、图像、视频能力在同一平台整合度高

风险：

- 需要你们先做一轮真实场景压测
- 当前项目里虽然保留了 MiniMax 支持，但主路径更偏向 Qwen

## 方案 C：商业化成本优先

- 实时主模型：`qwen-plus` 或 `MiniMax-M2.5`
- 批处理模型：`deepseek-chat`
- 复杂推理或特殊场景：高阶模型兜底

适合：

- 日活逐步扩大
- 要做周报、画像、标签、统计、复盘等后台任务

优点：

- 成本分层最清晰
- 便于后续做“不同任务走不同模型”的路由策略

风险：

- 需要新增 provider 适配层
- 运维与测试复杂度会上升

## 6. 价格与性价比观察

以下结论基于 2026-04-16 查询到的官方公开定价页面，价格后续可能调整，落地前应再次复核。

### 通义千问 / 阿里云百炼

从官方价格页可见，`qwen-plus` 在常见输入区间下价格明显低于高阶模型，适合作为默认在线模型；而更高阶的 `qwen3.6-plus`、`qwen-max` 更适合做升级通道，不适合作为所有请求的默认模型。

### DeepSeek

官方中文价格页显示，`deepseek-chat` 当前为：

- 百万 tokens 输入缓存命中：`0.2 元`
- 百万 tokens 输入缓存未命中：`2 元`
- 百万 tokens 输出：`3 元`

这使它非常适合作为低成本后台任务模型，例如：

- 周报草稿
- 情绪标签归纳
- 非实时摘要
- 运营分析初稿

### MiniMax

官方按量计费页显示，当前文本模型里：

- `MiniMax-M2.7` 输入 `2.1 元/百万 tokens`，输出 `8.4 元/百万 tokens`
- `MiniMax-M2.5` 输入 `2.1 元/百万 tokens`，输出 `8.4 元/百万 tokens`

从平台能力完整度看，MiniMax 更像一体化多模态平台；如果项目未来要走语音陪伴、数字人或多模态互动路线，它的战略价值会比“只看文本单价”更高。

## 7. WarmStudy 的推荐路由策略

建议把模型按任务分层，而不是“一把梭”：

### 一级：默认对话

适合场景：

- 学生日常倾诉
- 家长普通咨询
- 轻度情绪陪伴

建议模型：

- `qwen-plus` 或 `MiniMax-M2.5`

### 二级：高复杂对话

适合场景：

- 多轮长对话
- 家长深度沟通
- 需要更稳语义理解的场景

建议模型：

- `qwen-max` 或 `MiniMax-M2.7`

### 三级：离线分析

适合场景：

- 周报
- 摘要
- 标签
- 运营统计

建议模型：

- `deepseek-chat`

### 四级：高风险转人工辅助

适合场景：

- 自伤、自杀、极端家庭冲突等高风险线索

建议：

- 模型只做初步识别和结构化摘要
- 关键处置必须转人工或专业人员流程

## 8. 采购与成本控制建议

1. 不要把最高质量模型设成全量默认
2. 做按场景路由，而不是按用户等级路由
3. 优先优化 prompt、上下文裁剪和缓存，再谈换模型
4. 先观察 7 天或 30 天真实 Token 消耗，再决定大规模采购
5. 至少保留两个模型供应商方案，避免单点依赖

## 9. 建议写进答辩 PPT 的一句话

“我们采用分层模型路由策略：默认请求使用高性价比模型，复杂场景升级到高质量模型，后台批处理使用低成本模型，以兼顾体验、稳定性与长期运营成本。”

## 10. 后续工程建议

如果要把选型真正落地，建议补这三项：

1. 新增统一的 `model_provider` 适配层
2. 增加模型路由配置文件，而不是写死在业务逻辑中
3. 记录每次请求的模型、Token、时延、成功率与用户反馈
